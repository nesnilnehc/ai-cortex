# ADR 0002：plan-next v6 结构重构

**上下文**：`skills/plan-next/` v5.0.0；用户反馈 "思路是否清晰 / 逻辑是否自洽 / 用户是否看得懂"

## 背景

plan-next v5 在一年内迭代过 5 次（v1 矩阵 → v2 ADR-based → v3 准备门 → v4 状态机 → v5 两层输出），形成三层心智模型叠加：**矩阵**（缺口数据）→ **状态机**（聚焦解释）→ **两层输出**（显示权重）。三轮审视暴露四类问题：

1. **作者视角外泄**：S1-S7、G1-G4、P0-P3、Primary/Secondary、激活层、DONE/DEFERRED/ESCALATED 六套编号系统直接出现在对用户的报告里，认知负担高。
2. **结构性重复**：阶段 0.5（准备门）与阶段 0.6（S1 Greenfield）判据等价，同一事实被两套语言描述。阶段编号 0/0.5/0.6/1/1.5 是迭代补丁痕迹。
3. **逻辑冲突未解**：优先级规则（P0-P3）与状态机激活层是两套排序未合并；S7 Quiet mode 的 Secondary 写法违反"不用状态机折叠 Secondary"硬边界；矩阵宣称 MECE 但 S4/S5/S7 的 ceremony/静默/idle 不在矩阵内。
4. **输出顺序反认知**：当前顺序是 状态 → 扫描清单 → 准备门 → 路由，用户要读完半页事实罗列才看到行动。

问题不在局部疏漏，而在**作者视角与用户视角未分离**。继续打 5.x patch 无法解决术语体系、输出顺序、矩阵边界这些 breaking change。

## 决策

以下 5 条决策构成 v6.0.0 的核心改动。

### 决策 1：三步诊断结构替代小数点阶段

阶段 0 / 0.5 / 0.6 / 1 / 1.5 重构为 **扫（Scan）→ 诊（Diagnose）→ 荐（Recommend）** 三步。

- **扫**：资产清单（原阶段 0）
- **诊**：状态识别 + 缺口矩阵（原 0.5 并入；Rules×G1 / Why×G1 全空变为 S1 下的 short-circuit output mode，不再是独立阶段）
- **荐**：路由生成 + 分层展示（原阶段 1 + 1.5 合并）

### 决策 2：分离内部术语与用户输出术语

**内部术语（spec 保留，对用户不可见）**：S1-S7、G1-G4、P0-P3、激活层、Primary/Secondary

**用户输出术语（自然语言）**：

| 内部 | 对用户 |
|---|---|
| 项目状态 S2 Strategy drafted | "战略已起草，路线图尚未成型"（S2 作括号内追溯） |
| G1 / G2 / G3 / G4 | 缺 / 不全 / 漂移 / 错位 |
| P0 / P1 / P2 / P3 | 现在 / 下次 / 以后 / 可忽略 |
| Primary 激活层路由 | **现在该做** |
| Secondary 全矩阵压缩 | **其他可留意** |
| DONE / DEFERRED / ESCALATED | 完成条件 / 延后条件 / 上抛条件 |

用户报告里只呈现自然语言；内部编号仅作追溯标识出现在括号或诊断依据里。

### 决策 3：输出顺序按"行动优先"

旧顺序 `状态 → 输入源清单 → 准备门 → 路由` 改为：

```
# 下一步建议

## 现在该做（1-3 条，完整字段）

## 其他可留意（简短列表，≤5 条；超出折叠为 "+N more"）

## 诊断依据（末尾，可折叠）
- 项目情况：<一句自然语言摘要，括号附内部状态编号>
- 资产清单：<表>
- 判定规则：<状态识别 + tiebreak + 优先级>
```

"事实 / 诊断"降级为审计轨迹，"行动"提升为首屏。

### 决策 4：澄清矩阵 MECE 边界 + 形式化主从规则

**边界澄清**：缺口矩阵 MECE 覆盖所有**缺口型**发现；ceremony / 静默 / idle 是**非缺口**输出模式，由状态机独立产出。修正 v5 "矩阵 MECE 穷尽"措辞。

**主从规则**（新增显式规则）：

> 状态机决定**聚焦范围**（哪些格子进"现在该做"）；聚焦范围内按 P0 > P1 > P2 > P3 排序；聚焦外的发现进"其他可留意"。多状态匹配时取上游（S1 > S2 > ... > S7）。

这条规则在 v5 示例和注脚里隐含存在，v6 提升为显式规则。

**S7 Secondary 修正**：Secondary 无发现时写"无"，不再列 20 个"无"格。

### 决策 5：声明无状态 + 阈值集中化 + 位置定位

- **无状态声明**：硬边界加一行 "plan-next 是 stateless 的；每次调用从零开始重扫，不依赖上次结果。"
- **阈值集中**：`3 天 commit 静默` / `14 天漂移过期` / `5 条 backlog` / `低 confidence` 收入 frontmatter `defaults`，可被 input 覆盖。
- **位置定位**：SKILL.md 顶部新增 Positioning 小节（3-5 行）：plan-next 是**入口路由器**；单一维度问题（只查就绪度 / 只查漂移）用户可跳过它直接跑 `assess-docs` / `align-planning`。

## 替代方案

1. **继续打 5.x patch**：拒。术语体系、输出顺序、阶段编号都是 breaking，patch 会让结构更扭曲。
2. **只做输出层简化，不动内部**：拒。输出层自然语言要求内部也用对应结构作单一信源，否则漂移不可避免。
3. **废弃矩阵/状态机二选一**：拒。两者职责不同（矩阵是数据、状态机是解释），都保留但明确上下游关系即可。

## 后果

- **正面**：
  - 用户报告首屏即行动，术语密度降低 6→0 套内部编号
  - 阶段编号从 5 个小数点变为 3 个动词，记忆成本下降
  - 矩阵 MECE 声明不再失真；主从规则显式化，消除优先级/状态机冲突
  - Positioning 小节呼应"completeness via relational clarity"原则
- **负面**：
  - MAJOR bump 6.0.0；下游引用 plan-next 输出结构的协议需更新（目前无已知引用）
  - 示例全部重写
  - 作者维护 spec 时需"双语"：内部结构 + 用户自然语言映射
- **中性**：
  - 内部编号（S1-S7 / G1-G4）仍保留在 spec 的 Behavior 章节里，不是删除而是隔离作者视角
