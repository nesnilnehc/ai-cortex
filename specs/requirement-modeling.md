# 需求建模规范

> **数据契约**：定义需求文档的字段结构与校验规则
>
> 配套：质量评审见 [rules/requirement-quality.md](../rules/requirement-quality.md)

---

## 适用范围

适用于以下需求建模：

1. **功能需求**：新增功能、流程调整、体验优化
2. **非功能需求**：性能、安全、可维护性、可扩展性
3. **缺陷修复**：缺陷处理、性能优化、技术债偿还
4. **技术任务**：架构优化、依赖升级、基础设施改造

### 按需求类型的简化指引

| 类型 | 必填字段 | 可简化部分 |
|------|--------|---------|
| 功能需求 | 全部 7 个必填字段 | — |
| 缺陷修复 | 全部 7 个必填字段 | 背景（用问题陈述形式）、风险与约束（1 条关键风险；约束可选） |
| 技术任务 | 全部 7 个必填字段 | 背景（可用问题陈述形式） |
| 非功能需求 | 全部 7 个必填字段 | — |

**说明**：即便简化，7 个必填字段都必须完成。风险优先级与缓解策略**永不可选**。

---

## 7 个必填字段

> **条件性必填——范围定义**在下列情况下变为必填：
>
> - 需求涉及多系统或多模块集成
> - 需求与其他需求边界不清（容易被误读为超范围）
> - 技术任务或重构任务（必须明确变更边界）
> - 估算工作量 > 5 天
>
> 简单单功能需求或缺陷修复可省略范围定义，但建议至少列出 1–2 项 "out of scope" 以澄清排除项。

### 1. 需求 ID

**用途**：唯一标识需求，用于追溯与系统集成。

**格式**：`{PROJECT}-REQ-{nn}`（如 `ACME-REQ-05`、`MYAPP-REQ-12`）

**校验**：ID 唯一、格式正确、无重复。

**示例**：`APP-REQ-042`

---

### 2. 需求标题

**用途**：一句话概括需求的核心价值或问题。

**格式**：`[类型] 功能或问题描述`

**示例**：

```
[功能] 系统在需求生成中支持自动风险识别
[缺陷] 需求模板缺少可测试性验证字段
[优化] 将搜索延迟从 500ms 降至 <100ms
```

**校验**：标题 ≤ 80 字符，不含技术实现细节。

---

### 3. 背景与价值

**用途**：解释问题背景、期望状态与重要性——**不写解决方案或技术实现**。

**推荐格式**（用户故事）：

```
作为 [角色]
我希望 [期望的功能或结果]
以便 [价值或目的]
```

**替代格式**（问题陈述）：

```
背景：[现象或问题]
期望状态：[期望的结果]
影响：[受影响的用户或流程]
```

**示例**：

**示例 1（用户故事——推荐）**：

```
作为需求工程师
我希望有 API 能基于语义检索领域规范
以便 AI Agent 在生成需求时可获得相关规范上下文，从而提升需求质量
```

**示例 2（问题陈述）**：

```
背景：需求与代码提交未关联，评审记录分散
期望状态：需求与提交形成完整可追溯链路；评审历史集中
影响：需求管理、代码追溯、合规验证
```

**示例 3（反模式）**：

```
❌ 需要实现搜索 API 让 Agent 查规范
   （这描述了方案，不是背景）
```

**校验**：

- [ ] 不超过 300 字符
- [ ] 不含技术选型、实现路径或具体功能描述
- [ ] 清楚说明这条需求为何重要

---

### 4. 验收标准

**用途**：定义需求完成的可验证条件（功能 / 行为层面）。

**格式**：Gherkin BDD 或清单。

**示例 1（Gherkin）**：

```gherkin
场景：系统自动识别关键依赖

Given 需求工程师在生成 prompt 中配置了风险检测规则
And prompt 包含 "识别关键依赖"、"识别前置条件"、"识别集成风险"
When AI 生成需求初稿
Then 输出应包含独立的 "风险识别" 章节
And 风险章节至少包含 3 条已识别风险，每条含：
  - 风险描述
  - 概率
  - 影响等级
  - 优先级
  - 缓解策略
```

**示例 2（清单）**：

```
验收标准：
- [ ] 搜索接口已部署并能响应查询
- [ ] 查询响应时间 ≤ 500ms（p95）
- [ ] 返回相关条目 ≥ 3
- [ ] 相关性精度 ≥ 80%
- [ ] API 文档已发布，含输入/输出示例
- [ ] 集成测试覆盖 10+ 典型查询
```

**示例 3（缺陷修复）**：

```
验收标准：
- [ ] 需求模板新增 "可测试性验证" 字段
- [ ] 项目管理工具中显示该字段，位于 "验收标准" 之后
- [ ] 历史需求（>10 条）字段回填完成
- [ ] 文档更新含字段定义 + 3 条以上用例
- [ ] 团队反馈：新字段对评审有帮助、可无摩擦使用
```

**示例 4（非功能需求）**：

```
验收标准：
- [ ] API 响应时间 ≤ 500ms（p95，10M 数据集）
- [ ] 精度验证：top-3 相关性 ≥ 80%（50+ 典型查询）
- [ ] 已部署到生产并接入监控（延迟、精度、错误率）
- [ ] 性能基准报告发布，含前后对比
```

**校验**：至少 3 条验收标准，每条都可自动或人工验证。

**验收标准自检**：

- [ ] 无模糊形容词（如 "快"、"简单"、"友好"）→ 替换为可量化指标
- [ ] 每条标准可由第三方独立验证
- [ ] 非功能需求含具体数值（如延迟 ≤ 500ms、精度 ≥ 80%）

### 5. 依赖与前置条件

**用途**：记录需求的依赖关系与前置条件。

**格式**：

```
依赖需求：[需求 ID 或标题]
前置条件：[必须满足或具备的条件]
外部依赖：[第三方服务、工具、团队]
```

**示例**：

```
依赖需求：
  - APP-REQ-05（项目管理工具字段定制）
  - APP-REQ-08（需求评审清单）

前置条件：
  - 项目管理工具自定义字段已配置（status、review_links）
  - 需求模板 v1.0 已发布
  - Git Webhook 已配置

外部依赖：
  - 项目管理工具 API（需 API key）
  - Git 仓库（需 Webhook 权限）
```

**校验**：依赖列表无循环引用。

**循环依赖核查**：

- [ ] 本需求不被其依赖间接依赖（无环）
- [ ] 依赖链较长时，绘制依赖图核验

---

### 6. 风险、约束与假设

**用途**：提前识别风险（概率 × 影响）、确认约束、记录假设以待验证。

**格式**：

```
风险：[潜在问题]
  - 概率：[极低 / 低 / 中 / 高]（百分比）
  - 影响：[风险发生时对项目 / 交付的影响]
  - 优先级：[基于概率 × 影响，决定响应顺序]
  - 缓解：[降低风险的行动]

约束（已确认事实）：
  - 技术约束：[已验证的技术限制]
  - 时间约束：[已验证的时间限制]
  - 资源约束：[已验证的资源限制]

假设（待验证）：
  - [未验证的假设] | 验证方式 | 责任方
```

**示例**：

```
风险：
  - 搜索服务部署复杂、运维成本高
    - 概率：中（30%）
    - 影响：交付延迟 3–5 天 → 阻塞 Phase 2 KPI 验证
    - 优先级：高（必须提前处置）
    - 缓解：用 AI 自动生成 Dockerfile、K8s manifest、配置脚本

  - 搜索精度可能未达标（<80%）
    - 概率：低（15%）
    - 影响：需求质量下降 → 增加人工评审成本
    - 优先级：中（影响小但概率较高）
    - 缓解：第 2 周前完成 embedding 模型 POC，准备备选方案（model-A vs model-B）

约束（已确认）：
  - 技术：搜索服务选型（Technology-X）不可变；迁移成本 3–5 天
  - 时间：Phase 2 总工期 40 工日，不可延（Phase 3 已排期）
  - 资源：搜索流水线实施需技术团队 ≥ 2 人

假设（待验证）：
  - 项目管理工具 API 访问可获得 | 与运维团队确认 | 运维负责人
  - Embedding 模型 Model-A 在 10M 向量数据集上精度 ≥ 80% | 第 2 周前完成 POC 对比 | 技术负责人
```

**校验**：

- [ ] 功能需求 / 技术任务：至少 1 条风险、2 条约束
- [ ] 缺陷修复：至少 1 条关键风险；约束可选（如有则标注）
- [ ] 每条风险标注优先级（基于概率 × 影响）
- [ ] 假设附验证方式 + 责任方

**风险优先级计算**：优先级 = 概率 × 影响

- 高优先级：概率 ≥ 中 且 影响 ≥ 高
- 中优先级：（概率 = 中 且 影响 = 中）或（概率 = 高 且 影响 = 低）
- 低优先级：概率 ≤ 低 或 影响 ≤ 低

### 7. 需求来源

**用途**：澄清需求来源与上游决策上下文，便于理解业务价值与可追溯性。

**格式**：

```
来源类型：[功能请求 / 业务目标 / 故障 / 技术债 / 其他]
来源链接 / ID：[URL 或需求 ID]
父需求（如适用）：[关联需求 ID]
决策背景：[为何需要本需求的简要说明]
```

**示例**：

**示例 1 — 功能请求**：

```
来源类型：功能请求
来源链接：Issue #234（用户反馈：需求模板对可简化字段说明不清）
决策背景：Phase 2 用户反馈 → 改善模板可用性 → 降低撰写成本
```

**示例 2 — 业务目标**：

```
来源类型：业务目标
来源链接：2026 Q2 OKR：提升规范检索效率
父需求：无
决策背景：让 AI Agent 可靠地生成高质量需求
```

**示例 3 — 故障**：

```
来源类型：故障
来源链接：Phase 1 缺陷：需求模板缺可测试性字段
父需求：APP-REQ-02（需求模板增强）
决策背景：对 Phase 1 评审漏洞的回溯修复；防止同类问题复发
```

**校验**：

- [ ] 需求来源清晰可追溯（不是 "某人模糊提到"）
- [ ] 适用时标注父需求 ID
- [ ] 决策背景清晰（能向团队成员解释 "为什么做这个"）

## 可选字段（增强完整性）

### 8. 范围定义

**用途**：明确需求边界，防止范围蔓延。

**说明**：虽然列为可选，但符合上文 4 类条件之一时**必填**。

**格式**：

```
In Scope（本需求覆盖）：
  - [清晰列表]

Out of Scope（本需求不覆盖）：
  - [清晰列表]

最小可行版本（Walking Skeleton）：
  - [最薄的可交付形态]
```

**示例**：

```
In Scope：
- REST API 提供语义搜索，参数可配置（query、top_k、threshold）
- 与向量数据库集成做相似度计算
- API 文档（OpenAPI 规范 + 5+ 示例）
- 性能验证（p95 延迟 ≤ 500ms）

Out of Scope：
- 面向用户的搜索 UI（由 APP-REQ-25 处理）
- 搜索相关性精调（Phase 3 处理）
- 实时索引更新（推迟到 Phase 3）

最小可行版本：
- 基础 REST API（POST /search），接收查询文本
- 返回 top-3 结果含相似度
- API 文档含 2 条查询示例
```

---

### 9. 待解决问题

**用途**：记录评审 / 实施中尚未解决的事项，按是否阻塞分类。

**格式**：

```
[阻塞 / 非阻塞] [问题描述] | 责任方 | 计划解决方式
```

**示例**：

```
[阻塞] embedding 模型选型未定 | 技术负责人 | 第 2 周前完成 POC 对比、敲定选型
[非阻塞] 向量数据库限流策略 | DevOps 负责人 | 第 3 周前与厂商确认
[阻塞] API 鉴权方式（OAuth vs API Key）| 安全负责人 | 第 1 周前架构评审
```

**指引**：

- **阻塞问题**：未解决无法启动实施；必须在评审中处理或登记到依赖
- **非阻塞问题**：可与实施并行解决，但需追踪并确认

---

### 10. 关联文档与规范链接

**用途**：把需求与规范、参考标准关联起来。

**格式**：

```
关联规范：
  - [规范名]：[链接或路径]
  - [架构决策记录 ID]：[链接]

关联需求：
  - [其他关联需求 ID]
```

**示例**：

```
关联规范：
  - 技术栈规范（Node.js v18+、TypeScript）
  - API 安全规范
  - ADR-002：向量数据库选型

关联需求：
  - APP-REQ-12（知识库内容建设）
  - APP-REQ-18（向量化流水线实施）
```

---

### 11. 优先级与时间表

**格式**：

```
优先级：[P0 / P1 / P2]
  - P0：阻塞路径，必须完成
  - P1：关键路径，尽快完成
  - P2：非关键，可推迟

排期周期：[Week-1 / Week-2-3 / Phase-2-Week-1 等]
预估工作量：[x 工日]
```

---

### 12. 完成定义（DoD）

**用途**：衡量需求是否达到发布条件（流程与质量门）。**验收标准**回答 "功能完成了吗？"；**完成定义**回答 "可以发布了吗？"。

**格式**：

```
- [ ] 需求评审通过（评审人：___）
- [ ] 实施完成，PR 已提交
- [ ] 单元测试通过（覆盖率 ≥ 80%）
- [ ] 集成测试通过
- [ ] 代码评审通过
- [ ] 文档已更新
- [ ] 已部署到测试环境 / 生产环境
```

## 通用示例

### 示例 1：功能需求

```markdown
# 需求：知识库语义搜索接口

## 需求 ID
APP-REQ-15

## 需求标题
[功能] 实现知识库检索的语义搜索 API

## 背景与价值
作为系统集成方
我希望有 API 能对存储的规范执行语义搜索
以便外部系统能基于上下文高效检索相关文档

## 验收标准
- [ ] REST API `POST /search/semantic` 已实现且能响应查询
- [ ] 在 10M 条目数据集上响应时间 ≤ 500ms（p95）
- [ ] top-3 相关性精度 ≥ 80%（50+ 典型查询）
- [ ] API 文档完备（OpenAPI 规范 + 5+ 用例）

## 依赖与前置条件
依赖需求：APP-REQ-05（向量化流水线）、APP-REQ-08（向量数据库）
前置条件：知识库向量化完成（≥ 10M 向量）；向量数据库健康

## 风险、约束与假设
风险：高并发下 API 性能退化（中，35%）→ 第 1 周压测，引入缓存
约束：向量数据库选型不可变；阶段末（2 周）必须交付
假设：embedding 模型精度 ≥ 80% | 第 1 周基准测试 | 技术负责人

## 需求来源
来源类型：业务目标 | 2026 Q2 OKR：开放第三方与知识库的集成

## 范围定义
In Scope：REST API 实现、向量数据库集成、API Key 鉴权、OpenAPI 文档
Out of Scope：OAuth 鉴权、搜索 UI、实时向量更新
最小可行版本：POST /search，返回 top-5 结果含分数，含 2 条示例

## 优先级与时间表
P1 · Phase 2 第 3–4 周 · 4 工日

## 完成定义
- [ ] 单元测试覆盖率 ≥ 80%；集成测试通过
- [ ] 压测通过（100 QPS，p95 ≤ 500ms）
- [ ] API 文档已发布；已部署到 staging
```

### 示例 2：缺陷修复

```markdown
# 需求：新增可测试性验证字段

## 需求 ID
APP-REQ-02

## 需求标题
[缺陷] 需求模板缺少可测试性验证字段

## 背景与价值
作为 QA 工程师
我希望需求模板含可测试性验证项
以便验收标准的校验在评审中标准化

## 验收标准
- [ ] 需求模板新增 "可测试性验证" 字段，含 ≥ 5 项具体检查
- [ ] 项目管理工具中显示该字段并提供内嵌指引
- [ ] 已回填到 10+ Phase 1 历史需求
- [ ] 团队反馈：≥ 80% 评审人反映清晰度提升

## 依赖与前置条件
依赖需求：APP-REQ-03（需求模板 v1.0 基线）
前置条件：需求模板 v1.0 已部署到项目管理工具

## 风险、约束与假设
风险：新字段增加评审负担（中，40%）→ 提供自动化模板 + 30 分钟培训
约束：2 工日内完成（当前迭代）；1 名 QA 负责人
假设：清单覆盖 90% 常见问题 | 回顾 Phase 1 历史需求 | QA 负责人

## 需求来源
来源类型：故障 | Phase 1 复盘 Issue #127（验收标准歧义）
父需求：APP-REQ-03

## 优先级与时间表
P1（阻塞 Phase 2 需求评审）· Phase 2 第 1 周 · 2 工日

## 完成定义
- [ ] 清单设计评审通过；项目管理工具字段已配置
- [ ] Phase 1 需求已更新（10+ 条）；团队培训完成
```

## 附录：术语表

| 术语 | 定义 |
|------|------|
| **验收标准（Acceptance Criteria）** | 定义需求完成时的具体、可度量条件 |
| **背景与价值（Background & Value）** | 解释问题或机会的上下文，不提出解决方案 |
| **BDD（行为驱动开发）** | 用 Gherkin 语法（Given-When-Then）表达行为的测试方法 |
| **完成定义（DoD, Definition of Done）** | 需求可发布前必须满足的流程 / 质量门清单 |
| **依赖（Dependency）** | 必须先存在 / 完成的需求或资源 |
| **Gherkin** | 编写可执行规范的结构化语言（Given-When-Then 格式） |
| **非功能需求（Non-Functional Requirement）** | 描述系统性质（性能、安全、可扩展性）而非功能的需求 |
| **P0 / P1 / P2** | 优先级（P0 = 阻塞，P1 = 关键，P2 = 非关键） |
| **前置条件（Prerequisite）** | 启动工作前必须满足的条件或资源 |
| **风险优先级（Risk Priority）** | 概率 × 影响算出，决定缓解的紧迫度 |
| **范围定义（Scope Definition）** | 明确需求所含 / 不含的边界 |
| **用户故事（User Story）** | 需求格式："作为 [角色] 我希望 [功能] 以便 [价值]" |
| **Walking Skeleton** | 演示核心功能的最小可行实现 |

## 关联标准与参考

本规范兼容并借鉴：

- **IEEE 830**：软件需求规格说明的 IEEE 推荐实践
- **敏捷宣言**：强调清晰需求胜于详尽文档
- **Gherkin / Cucumber**：验收标准的结构化 BDD 格式
- **风险管理标准**：ISO 31000 风险识别与优先级
- **SWEBOK**：软件工程知识体系中的可追踪性最佳实践
